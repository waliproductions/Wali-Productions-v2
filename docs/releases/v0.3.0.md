# Release Notes — v0.3.0

**Release date:** 2026-06-27
**Tag:** `v0.3.0`
**Branch:** `main`

---

## Overview

v0.3.0 delivers three major additions to the Wali Productions admin portal:

1. **Production Authentication** — iron-session sealed cookie sessions with bcryptjs password verification, a `/login` page, and automatic protection of all `/admin/*` routes via the Next.js 16 proxy layer.
2. **Government Operations Center** — a full hub-and-spoke admin section across seven pages for managing capability statements, registration status, technical capabilities, NAICS/PSC codes, past performance, and government documents.
3. **Admin Portal Foundation** — portfolio management, analytics dashboard, contact submission dashboard, audit log viewer, and a system settings page — all config-driven and read-only.

---

## What's New

### Authentication

- **Login page** (`/login`) with a Server Action form powered by `useActionState` (React 19)
- **iron-session v8** sealed cookie sessions — AES-GCM encryption via the Web Crypto API (Edge Runtime compatible)
- **bcryptjs v3** password verification — pure JS, no native bindings, Hostinger-safe
- **Timing-safe login** — bcrypt comparison always runs, even when credentials are missing or the username doesn't match, to prevent timing-based enumeration
- **Session cookie** `__admin_session` — HttpOnly, Secure (production), SameSite=Lax, 7-day TTL
- **Proxy route protection** at `src/proxy.ts` — all `/admin/*` routes redirect to `/login` when unauthenticated
- **Logout** via a Server Action form — no API route, no JavaScript required
- **Username display** in the admin header; accessible without reading session values on the client
- **Role field** (`role?: "admin"`) reserved in session type for future RBAC

### Government Operations Center

- **Hub dashboard** (`/admin/government`) with readiness estimate, navigation cards, and registration overview
- **Capability Statement** (`/admin/government/capability-statement`) — proposal-ready preview of competencies, differentiators, and capability paragraphs
- **Registration Tracker** (`/admin/government/registration`) — all six registration items with Pending/Verified status
- **Technical Capabilities** (`/admin/government/capabilities`) — 8 current domains, 36 skills, 10 future domains
- **NAICS Codes** (`/admin/government/naics`) — pending state with policy guidance
- **PSC Codes** (`/admin/government/psc`) — pending state with policy guidance
- **Past Performance** (`/admin/government/past-performance`) — 8 performance categories; pending state with authorization policy
- **Document Browser** (`/admin/government/documents`) — server-reads `docs/02-Government/*.md` with document selection via URL params

### Admin Portal Foundation

- **Portfolio management** — filterable table, detail pages, config-driven project entries
- **Analytics dashboard** — contact volume trends, pipeline state, service breakdown, recent audit events; all derived from local filesystem
- **Contact submission dashboard** — inbox view, search and filter, individual submission detail, CSV export
- **Audit log viewer** — searchable event feed with expandable detail
- **Settings page** — env var presence diagnostics, storage directory health, pipeline health, government readiness summary, system info

## Upgrade Notes

### Migration: `ADMIN_PASSWORD` → `ADMIN_PASSWORD_HASH` + `SESSION_SECRET`

v0.3.0 removes support for the plaintext `ADMIN_PASSWORD` environment variable. Two new environment variables are required.

**1. `ADMIN_PASSWORD_HASH`** — bcryptjs hash of your admin password

Generate the hash:
```bash
node -e "require('bcryptjs').hash('your-password', 12).then(h => console.log(h))"
```

**2. `SESSION_SECRET`** — iron-session encryption key (minimum 32 characters)

Generate a secure secret:
```bash
openssl rand -base64 32
```

**3. `ADMIN_USERNAME`** — unchanged from earlier usage, but now required at startup

**Your updated `.env.local` should contain:**
```bash
ADMIN_USERNAME=admin
ADMIN_PASSWORD_HASH=$2b$12$<your-generated-hash>
SESSION_SECRET=<your-generated-secret>
```

If any of the three are missing, `/admin` routes remain inaccessible until they are set. The `/admin/settings` Authentication card shows which variables are configured.

**Remove `ADMIN_PASSWORD`** from your environment — it is no longer read.

## Developer Notes

### Compatibility

- **Node.js** ≥ 18.18 required (Web Crypto API, `node:fs/promises`)
- **Next.js 16** — this release uses `src/proxy.ts` with `export async function proxy()` (Next.js 16 convention). The `src/middleware.ts` / `export function middleware()` convention from Next.js 13–15 is not used.
- **React 19** — `useActionState` replaces the deprecated `useFormState`
- **iron-session v8** — v7 and below have an incompatible API; see the iron-session changelog if upgrading from v7

### Systems Unchanged in v0.3.0

The following systems are untouched and fully compatible:

- Public site pages (`/`, `/government`, `/portfolio`, `/contact`)
- Contact form and API route (`/api/contact`)
- SMTP pipeline (Nodemailer)
- Contact filesystem storage (`app-data/contact-submissions/`)
- Audit logging
- Analytics filesystem reads
- `src/config/government.ts` — no structural changes

### Deployment (Hostinger)

1. Add the three new environment variables (`ADMIN_USERNAME`, `ADMIN_PASSWORD_HASH`, `SESSION_SECRET`) in the Hostinger control panel before deploying.
2. `bcryptjs` is pure JavaScript — no native compilation step required on the server.
3. `iron-session` uses the Web Crypto API via `crypto.subtle` — available in Node.js 18.18+ without flags.
4. No database, Redis, or external session store is required. Sessions are sealed into the cookie.
5. `docs/02-Government/` must be present in the deployed artifact for the document browser to render content.
